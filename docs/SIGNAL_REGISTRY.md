# Signal registry

## What is it?

A PostgreSQL table (and its Python wrapper in
[`src/signals/registry/signal_registry.py`](../src/signals/registry/signal_registry.py))
that tracks **every signal ever created** — its name, version, status, who built it, when
it was validated, its current IC, and whether it is in production. The registry is the
single source of truth for what is live.

## Signal lifecycle

```
DISCOVERED → VALIDATED → STAGING → PRODUCTION → RETIRED
```

| Status | Meaning |
|--------|---------|
| `DISCOVERED` | Generated by an AI agent or built by hand. Not yet validated. |
| `VALIDATED`  | Passed all 7 validation tests (see [TESTING.md](TESTING.md)). Ready for combination. |
| `STAGING`    | In the optimal-weights combination but **paper trading only**, for 30+ days. |
| `PRODUCTION` | Live, real money, monitored daily. |
| `RETIRED`    | IC decayed below 0.01 for 60+ consecutive days. Removed from production. |

Transitions are explicit and timestamped. Code is **never deleted** on retirement — only
the status changes.

## Table schema

```sql
CREATE TABLE signals (
    id              SERIAL PRIMARY KEY,
    name            VARCHAR(100) UNIQUE NOT NULL,
    version         VARCHAR(20) NOT NULL,       -- semver, matches SignalScore.signal_version
    status          VARCHAR(20) NOT NULL,       -- SignalStatus enum value
    description     TEXT NOT NULL,
    category        VARCHAR(50),                -- momentum, quality, value, earnings, ...
    data_sources    TEXT[],                     -- FactSet datasets used
    point_in_time   BOOLEAN NOT NULL,           -- confirmed look-ahead safe?
    created_at      TIMESTAMP DEFAULT NOW(),
    validated_at    TIMESTAMP,
    promoted_at     TIMESTAMP,
    retired_at      TIMESTAMP,
    mean_ic         FLOAT,
    icir            FLOAT,
    sharpe          FLOAT,
    holding_period  INT,                        -- days
    universe        VARCHAR(100),               -- e.g. SP500, Russell2000
    notes           TEXT
);
```

## Rules

- `name` is unique and `snake_case`; `version` follows semver and must match the
  `signal_version` stamped on that signal's `SignalScore` outputs.
- A signal cannot move to `VALIDATED` without a stored `BacktestResult` where
  `passed_validation = true`.
- A signal cannot move to `PRODUCTION` without ≥30 days in `STAGING` with live IC above
  threshold.
- `point_in_time = true` is mandatory before any backtest use (PRINCIPLES.md Rule 8).

The table is seeded by `scripts/seed_signal_registry.py` (`make setup-db`).
